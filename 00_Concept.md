## 为什么做这件事

过去一年，AI Agent 的研究和应用已经进入快速发展阶段，涌现出了很多优秀的Agent系统，其中包括Claude Code、Codex、Cursor等以Coding为主的Agent，也有包括OpenClaw、HermesAgent等以通用性的个人助理Agent，此外还有包括Manus在内的一系列自主执行Agent系统，上述的Agent系统都在各自的领域取得了相当出色的进展和广泛的应用。

然而，当我们回过头来看着一切，我不禁开始思考，虽然目前AI Agent系统的能力已经取得了不错的成效，但是他们真的就已经是最优解了吗？

作为一名科研工作者，AI的出现以及Agent能力的提升在一定程度上解放了我的生产力，从2025年下半年开始，我陆续使用了Cursor、Claude Code、Codex、OpenClaw、HermesAgent等Agent系统，协助我完成了大量的代码编写、数据分析处理、文档报告撰写等工作。但在这个过程中我发现了一些非常影响用户使用的问题：


### 1. Agent单次任务有效执行能力限制

最开始使用Agent系统的时候，我们都有一个非常美好的愿景，就是我给Agent一个任务，Agent就能够独立完成这个任务，不需要我过多的干预和指导。
然而在实际使用中我们发现与Agent工作的方式对于最终的交付质量影响极大，因为Agent在单次对话中能执行的任务能力是有限的，当在一次对话中让Agent执行过于复杂的任务，虽然Agent可能会通过拆解逐步完成，但是每一步都会略微产生偏离，导致最终的交付结果和预期有较大差距。
这类问题在代码生成、数据分析、报告撰写、文献整理、科研实验规划、工程方案论证等任务中都非常常见。因此我们不得不人工将一个任务拆解成若干个子任务，逐步指挥Agent完成，不断告诉 Agent：

```text
先做第一步。
继续第二步。
按照刚才的计划继续。
这里不对，重新来。
这一步需要补充证据。
继续后面的部分。
```
这虽然能够提升结果质量，但本质上是把 Agent 的任务管理负担转嫁给了用户。用户不仅要知道最终目标，还要知道如何拆解任务、如何检查结果、如何修正偏差、如何维持上下文连续性。

这说明当前 Agent 的核心问题不是“不会执行”，而是：**缺少一种稳定承载复杂任务执行过程的外部结构。**


### 2. 复杂任务自主规划执行的信任危机

实际上Agent的单次任务执行能力限制大家都已经意识到，而Manus、OpenClaw、HermesAgent等自主执行Agent系统也是为了解决这一问题，通过自主规划然后逐步引导的方式让Agent能够完成更为复杂的任务，但是把一个任务完全交给Agent进行自主规划真的是我们想要的吗？
在轻量办公、资料整理、个人事务处理、常规内容生成等场景中我认为是没有问题的，但是如果将Agent真正应用到生产场景中呢？Agent的每一次动作都是应该被严格约束并审查的，我们开始面临新的问题：

```text
Agent 为什么这么规划？
规划执行路线是否合理？
它是否遗漏了关键约束？
它调用工具之前是否应该先经过确认？
它生成的结果是否有依据？
它的中间判断是否可追溯？
它失败后为什么选择这个修复方向？
```
对于工业、科研、工程、合规、项目交付等任务来说，Agent 的每一次动作都不应该是黑盒行为，因此我们需要一种新的方式来让Agent能够在用户的指导下完成复杂任务的规划和执行，而不是完全交给Agent进行自主规划执行。
因此，Manus的 “Less structure, more intelligence” 显然并不适合所有应用场景，对于真正的生产和工程任务，我们更需要的是：“More structure, more reliability”，通过合理规划与执行约束提高交付的可靠性。

### 3. Harness Engineering 实战的尴尬处境

在前两种问题背景下，Harness Engineering被提出来，其本质是一种工程化的工作方法：认为设计任务计划、阶段目标、执行规则、审查标准等，让Agent按照结构化的说明逐步执行，通过项目记忆引导Agent更稳定地完成复杂任务。

但问题在于，当前 Harness Engineering 很大程度上依赖用户个人能力，对于大多数人的使用要求过高。用户需要自己理解并制定计划、约束、项目记忆、执行规范、审查标准等，还要在每个阶段之后人工告诉 Agent 继续执行下一阶段。这导致一个非常尴尬的使用体验：

```text
用户不是在享受 Agent 的自动化能力，
而是在扮演 Agent 的项目经理、审查员和调度器。
```

这也是为什么很多用户在使用复杂 Agent 工具时，会反复输入：

```text
按照计划继续执行。
Continue。
继续完成下一步。
根据刚才的计划往下做。
```

虽然Claude Code/Codex等推出了 /Goal 模式，但在实际使用中 /Goal 的执行我们也总是会担心他会不会在某一个阶段趁我不注意产生偏离（又重复引入了自主规划阶段的信任危机）。

因此我们真正需要产品化的并不是一个Prompt、一个Skill，而是：**能把Harness Engineering从个人经验升级为系统内的自动执行Workflow能力。**

### 4. Plan-Act-Review的模型自闭环运行

先规划后执行最终审查评价是目前公认的Agent工作执行模式。但是在面对长流程的任务执行过程中，仍有问题需要面对：

- 什么时候进行Review？
- 每个阶段的Review标准是什么？
- Review不通过怎么调整后续执行路线？

实际上目前大多数的Review可以由模型自己完成，并且在规划Harness的过程中也会给出每个阶段的评价标准，但是针对Review不通过情况下的处理执行依旧需要人工的介入来对后续的模型行为进行调整和修正。

因此我们不仅需要让Agent能够按照预期的Workflow自动化执行，更重要的是：**让Agent能够 Plan-Act-Review-Repair 的执行反馈闭环。**

### 5. 不同模型能力差异导致的交付不稳定

现实中，不是所有用户都能使用 GPT、Claude Opus、等顶级闭源模型。很多用户可能只能使用中等能力模型、本地模型、企业私有模型或成本受限模型。不同模型在长上下文理解、复杂推理、结构化输出、工具调用、事实准确性、自我修复能力等方面差距很大。

如果一个系统完全依赖单个强模型的能力，那么它在强模型环境下表现很好，在普通模型环境下则容易崩溃。

因此通过Workflow的结构化能力，尽可能缩小不同模型之间的交付差异。通过任务拆解、节点契约、上下文包、证据包、结构化输出、独立审查、失败修复、人工检查点等机制，让普通模型在一个受控流程中完成原本只有强模型才能稳定完成的复杂任务。

## 我们需要什么样的Agent

当我们面对结构设计—仿真—优化迭代、科研项目调研—实验—数据分析—报告撰写、项目资料整理—证据链构建—验收材料生成等长流程任务时，真正需要的不是一个聊天 Agent，也不是一个完全自主行动的黑盒 Agent，而是：

- 可编排控制的 Agent 工作流水线
- 自动化构建的 Harness 系统
- 人工控制下的 Review-Repair 自反馈闭环
- 可视化、可审查、可回滚的执行工作台
- 补偿模型能力差异的认知型 Workflow

这就是 Cognitive Workflow 要解决的问题。

## 我们怎么做

Cognitive Workflow 的核心思想是： **把复杂任务从一次性 Agent 对话，转化为可编排、可验证、可复现、可修复的认知型工作流。**

Workflow不是简单地把 LLM 节点、工具节点和条件节点连接起来，而是一套面向复杂认知任务的执行协议：

- 任务被拆解为节点
- 节点拥有明确输入、输出、上下文、证据、模型、Skill 与审查规则
- 执行结果必须经过评价
- 评价不通过会进入修复回路
- 关键节点可以暂停等待人工确认
- 任务执行通过项目内Harness严格控制
- 任务全过程被记录、可回滚、可复用


## 我们怎么做

Cognitive Workflow 的核心思想是：

> **把复杂任务从一次性 Agent 对话，转化为可编排、可验证、可复现、可修复的认知型工作流。**

它不是简单地把 LLM 节点、工具节点和条件节点连接起来，也不是把任务完全交给 Agent 自主规划。它更像是一套面向复杂认知任务的执行协议：

```text
任务被拆解为节点；
节点拥有明确输入、输出、上下文、证据、模型、Skill 与审查规则；
执行结果必须经过评价；
评价不通过会进入修复回路；
关键节点可以暂停等待人工确认；
全过程被记录、复盘和沉淀为可复用模板。
```

### 1. 两种 Workflow 创建方式

Cognitive Workflow 提供两种 Workflow 创建模式。

第一种是 **AI 自动规划编排**。用户打开 Workflow 规划模式后，通过对话输入任务目标、背景信息、附件资料和必要上下文。系统不会立即生成 Workflow，而是依次完成：

```text
探索
→ 理解
→ 必要时澄清确认
→ 草案编排
→ 草案校验与修复
→ 可视化预览
→ 用户确认创建
```

其中：

- **探索**：整理用户历史对话、任务目标描述、附件、参考资料、项目记忆等已有上下文；
- **理解**：判断任务目标、边界、难度、输入资料、输出要求是否足以支撑规划；
- **澄清确认**：仅在缺少关键决策信息时触发，采用问答式确认，每次提供 3 个按优先级排序的备选方案，并支持用户自定义输入；
- **草案编排**：生成结构化 Workflow 草案；
- **校验修复**：检查草案格式、节点完整性、边关系、条件分支、评价节点、修复回路是否可执行；
- **预览确认**：通过 Node Canvas 可视化展示 Workflow 草案，用户可直接创建，也可继续通过自然语言修改。

第二种是 **手动节点编排**。用户可以直接进入 Node Canvas 编辑器，手动添加节点、连接节点、配置每个节点的 Prompt、模型、上下文、附件、Skill、MCP、输入输出和审查规则。

这两种方式不是互斥关系。用户可以：

```text
先由 AI 自动规划生成草案，
再进入手动节点编辑器微调；
也可以先手动搭建 Workflow，
再让 AI 根据当前图结构提出优化建议。
```

### 2. 三类核心节点

Cognitive Workflow 中的节点不只是普通任务步骤，而是具有明确职责的认知任务单元。

#### 2.1 执行 Task

执行 Task 用于完成具体任务，例如读取资料、提取信息、调用工具、生成内容、执行代码、运行仿真、整理表格、生成报告等。

执行 Task 需要定义：

```text
节点目标
输入数据
输出数据
执行 Prompt
上下文依赖
外部 Reference
使用模型
加载 Skill
调用 MCP / 工具
自查规则
失败重试策略
```

#### 2.2 评价 Task

评价 Task 用于对某个执行 Task 的输出进行独立审查、测试或质量判断。

评价 Task 需要定义：

```text
审查对象
审查范围
评价标准
通过条件
不通过条件
通过后的下一个节点
不通过后的下一个节点
最大重试次数
失败诊断输出
```

评价 Task 的输出不应只有“通过 / 不通过”，而应包括：

```text
失败原因
失败类型
未满足规则
严重程度
建议修复方式
建议回流节点
是否需要人工介入
```

#### 2.3 修复 Task

修复 Task 用于处理评价不通过后的改进逻辑。它不是简单重试，而是根据失败诊断生成修复补丁。

修复 Task 可能修改：

```text
执行 Prompt
输出 Schema
上下文包
证据包
节点输入
审查规则
Workflow 结构
执行模型
```

例如：

```text
评价 Task 判断“缺少证据引用”
→ 修复 Task 为执行节点增加 evidence_id 输出要求
→ 执行 Task 重试
→ 评价 Task 再次审查
```

### 3. Node Contract：节点契约

为了降低模型自由发挥带来的不确定性，每个节点都必须被契约化。

一个节点不是一句自然语言任务，而应包含：

```text
目标
输入 Schema
输出 Schema
上下文要求
证据要求
可调用工具
可使用模型
执行提示词
审查规则
失败策略
```

节点契约的作用是把模型执行从“开放式回答”变成“受控任务执行”。

---

### 4. Context Pack 与 Evidence Pack

不同模型对长上下文的处理能力差异很大。为了减少模型负担，系统不应把所有资料直接塞给模型，而应在每个节点执行前构建 **Context Pack**。

Context Pack 包含：

```text
当前节点目标
全局任务摘要
用户约束
项目记忆
上游节点输出
相关参考资料片段
输出格式要求
审查规则
```

对于事实性、研究性、报告性任务，系统还应构建 **Evidence Pack**。

Evidence Pack 包含：

```text
证据编号
证据内容
来源文件
页码 / 段落 / 链接
可信度
相关性评分
支持的结论
冲突证据
```

模型生成内容时必须基于 Evidence Pack，而不是凭空生成。

---

### 5. Review-Repair 闭环

Cognitive Workflow 的核心不是“节点执行”，而是“节点执行后的质量闭环”。

标准流程为：

```text
执行 Task
→ 评价 Task
→ 通过：进入下一个节点
→ 不通过：进入修复 Task
→ 修复后重新执行
→ 再次评价
→ 多次失败后人工介入
```

这使得 Agent 执行不再是黑盒、一次性、不可控的过程，而是一个可持续改进的闭环。

LangGraph 的 interrupt 机制也体现了类似思想：当需要外部输入时，可以暂停图执行，保存当前状态，并在用户提供输入后恢复执行。这样的持久化与人类介入能力，是复杂 Agent 工作流走向生产化的重要基础。

### 6. Workflow 项目记忆与执行状态记录

Cognitive Workflow 需要自动构建项目级 Harness 记忆，包括：

```text
任务目标
用户约束
关键决策
当前 Workflow 结构
节点执行状态
节点输出产物
审查结果
失败修复历史
人工确认记录
最终交付物
```

同时，每次节点执行、审查、修复、回滚都应被记录下来，支持：

```text
继续执行
失败重试
版本回滚
历史对比
失败复盘
Workflow 模板沉淀
```

### 7. 可视化 Workflow 工作台

Cognitive Workflow 的交互核心是一个 Node Canvas 工作台。

用户可以看到：

```text
Workflow 图结构
节点状态
执行路径
通过 / 不通过分支
重试回路
修复回流
当前运行节点
右侧节点详情
底部流式输出
底部自然语言 Chat Box
```

用户不是通过连续输入“继续执行”来控制 Agent，而是通过：

```text
点击运行
查看节点
审查结果
自然语言修改节点
确认修复方案
继续 Workflow
```

来完成复杂任务控制。

### 8. 模型能力补偿层：Model Capability Compensation Layer

Cognitive Workflow 还需要承担一个更深层的任务：通过系统结构尽量缩小不同模型之间的能力差异。

我们可以将这一层定义为：

> **Model Capability Compensation Layer，模型能力补偿层。**

它的目的不是把普通模型变成强模型，而是通过任务拆解、上下文约束、结构化输出、证据边界、审查节点和修复节点，让普通模型在一个稳定的执行框架中完成更高质量的交付。

模型能力补偿层包括：

```text
Task Decomposition：将复杂任务拆成小节点
Node Contract：限定每个节点的输入输出和成功标准
Context Pack：给模型提供当前节点所需的最小充分上下文
Evidence Pack：为事实性输出提供证据边界
Structured Output：通过 Schema 约束输出结构
Evaluation Task：独立审查节点输出
Repair Task：审查不通过后生成修复补丁
Human Checkpoint：在关键节点暂停等待人工确认
Reflection Memory：记录失败与修复经验，沉淀为后续模板
```

它的核心思想是：

> **强模型依赖单次综合能力，Cognitive Workflow 依赖系统化认知脚手架。**

## Cognitive Workflow 是什么

Cognitive Workflow 是一种面向复杂认知任务的 Agent 工作流执行框架。它将复杂任务拆解为一组带有明确输入、输出、上下文、证据、模型、工具、审查规则和失败策略的任务节点，并通过执行、评价、修复、人工确认和状态记录机制，使 Agent 能够以可控、可验证、可复现的方式完成长流程任务。

一句话定义：

> **Cognitive Workflow 是一种把复杂任务转化为可执行、可审查、可修复、可复现 Agent 工作流的认知型执行协议。**

它不是传统自动化工作流，也不是单纯的 Agent 自主执行。它强调的是：

```text
结构化任务拆解
节点契约化执行
证据驱动生成
独立评价审查
失败闭环修复
人工可控介入
全过程状态记录
跨模型交付一致性
```

### 1. Cognitive Workflow 的基本结构

一个 Cognitive Workflow 通常包含：

```text
Start 节点
执行 Task 节点
评价 Task 节点
修复 Task 节点
人工确认节点
条件边
重试边
失败边
结束节点
```

示例：

```text
用户任务
→ 任务理解
→ 资料解析
→ 证据检索
→ 方案生成
→ 方案审查
→ 不通过则修复
→ 通过则生成报告
→ 报告审查
→ 导出交付
```

其中，关键不是“流程长”，而是每个节点都具有明确契约，每个关键结果都可以被审查，每次失败都能进入可解释的修复路径。

### 2. Cognitive Workflow 的核心对象

#### 2.1 Workflow

Workflow 是一个有向图，而不是简单列表。

它允许：

```text
顺序执行
条件分支
并行执行
失败回流
循环重试
人工中断
子 Workflow
```

#### 2.2 Node

Node 是 Workflow 的最小执行单元。每个 Node 都必须拥有明确职责。

核心节点类型包括：

```text
Execution Node
Evaluation Node
Repair Node
Human Checkpoint Node
Tool Node
Memory Node
Artifact Node
```

#### 2.3 Edge

Edge 表示节点之间的执行关系。

常见边类型包括：

```text
normal：普通流转
pass：审查通过
fail：审查不通过
retry：重试
repair：进入修复
human：等待人工确认
optional：可选路径
loop：循环回流
```

#### 2.4 Artifact

Artifact 是节点执行产物，包括：

```text
中间文件
报告章节
表格
代码
测试结果
仿真结果
证据包
审查报告
最终交付物
```

#### 2.5 Memory

Memory 是当前项目被确认的上下文、约束、决策与历史执行记录，不等于聊天历史，也不等于参考资料库。

#### 2.6 Reference

Reference 是当前 Workflow 执行时可检索、可引用的外部资料，包括文档、代码库、数据表、历史项目、网页、工具输出等。

---

### 3. Cognitive Workflow 的运行生命周期

Cognitive Workflow 的生命周期可以划分为六个阶段。

#### 阶段一：任务输入

用户输入任务目标、背景、附件和约束。

#### 阶段二：Workflow 规划

系统探索已有上下文，理解任务目标，必要时进行澄清确认，并生成 Workflow 草案。

#### 阶段三：Workflow 确认

用户预览 Workflow，可以直接创建，也可以自然语言修改或进入手动编辑器调整。

#### 阶段四：Workflow 执行

系统按照节点图执行任务，每个节点按契约读取输入、调用模型和工具、生成输出。

#### 阶段五：审查与修复

评价节点对执行结果进行审查。不通过时进入修复节点，生成修复补丁并重新执行。

#### 阶段六：交付与沉淀

最终产物生成后，系统归档产物、记录版本、更新项目记忆，并可将成功流程沉淀为 Workflow 模板或 Skill。

---

### 4. Cognitive Workflow 的核心能力

#### 4.1 可规划

系统能从模糊任务中自动生成可执行 Workflow 草案。

#### 4.2 可控制

用户可以查看、修改、确认每个节点和执行路径。

#### 4.3 可审查

每个关键输出都可以被独立评价 Task 检查。

#### 4.4 可修复

审查不通过不会导致任务失败，而会进入修复闭环。

#### 4.5 可复现

每个节点的输入、输出、模型、Prompt、证据、工具调用和审查结果均被记录。

#### 4.6 可迁移

同一个 Workflow 可以在不同模型、不同项目、不同资料条件下复用。

#### 4.7 可补偿模型差异

通过任务拆解、节点契约、上下文包、证据包、审查节点和修复节点，系统尽可能降低不同模型能力差异对最终交付质量的影响。

#### 4.8 可沉淀

成功 Workflow 可以被保存为模板，失败修复经验可以被沉淀为 Skill，项目上下文可以形成可复用 Harness。

---

### 5. Cognitive Workflow 的技术抽象

Cognitive Workflow 可以理解为三层结构。

#### 5.1 任务层

任务层负责定义用户要完成什么，包括：

```text
任务目标
业务背景
约束条件
交付物
成功标准
风险边界
```

#### 5.2 工作流层

工作流层负责把任务拆解为图结构，包括：

```text
节点
边
条件分支
评价节点
修复节点
人工检查点
```

#### 5.3 执行运行时层

运行时层负责具体执行，包括：

```text
模型调用
工具调用
上下文包构建
证据包构建
输出校验
审查判断
失败修复
状态持久化
```

---

## Cognitive Workflow 和其他 Agent 有什么不同

Cognitive Workflow 与现有 Agent 系统的根本区别在于：

> **它不是让 Agent 自主完成任务，也不是简单搭建 AI 应用流程，而是把复杂任务变成可验证、可修复、可复现的认知型执行图。**

---

### 1. 与 Dify 的区别

Dify 的 Workflow 强调通过拖拽界面和无限画布连接不同节点，构建 AI 应用流程。其官方介绍中提到 Workflow 的核心是直观的拖拽界面，用户通过连接不同节点来构建应用；Dify 的 Agent Node 则允许 LLM 自主控制工具，并动态决定何时调用工具完成复杂任务。

Cognitive Workflow 与 Dify 的差异在于：

| 对比项 | Dify Workflow | Cognitive Workflow |
|---|---|---|
| 核心目标 | 构建 AI 应用流程 | 执行复杂认知任务 |
| 主要对象 | LLM 节点、工具节点、应用链路 | 执行 Task、评价 Task、修复 Task、人工检查点 |
| 重点 | 应用编排与发布 | 过程可控、质量审查、失败修复 |
| 失败处理 | 节点失败或应用调试 | 结构化诊断、修复补丁、回流重试 |
| 适合场景 | AI 应用、RAG、客服、内容生成 | 科研、工程、技术论证、报告、仿真、合规、项目交付 |

一句话：

> **Dify 更像 AI 应用构建平台，Cognitive Workflow 更像复杂任务执行操作系统。**

---

### 2. 与 Manus 的区别

Manus 的官方表达是 “Less structure, more intelligence”，即希望通过更少的结构，让 Agent 依赖更强的智能自主完成任务。

Cognitive Workflow 选择的是相反方向：

> **More structure, more reliability.**

| 对比项 | Manus | Cognitive Workflow |
|---|---|---|
| 核心理念 | 更少结构，更多智能 | 更多结构，更高可靠性 |
| 运行方式 | Agent 自主规划执行 | 用户可控的 Workflow 图执行 |
| 用户信任来源 | Agent 能力 | 节点契约、审查、修复、记录 |
| 适合场景 | 泛化任务、个人事务、自动执行 | 高风险、长流程、需审查任务 |
| 失败处理 | Agent 自主调整或用户追问 | 评价节点诊断 + 修复节点闭环 |

一句话：

> **Manus 强调把任务交给 Agent，Cognitive Workflow 强调把 Agent 放进可控流程。**

---

### 3. 与 Claude Code / Codex 的区别

Claude Code 和 Codex 都是 Coding Agent。Claude Code 可以理解代码库、编辑文件、运行命令，并与终端、IDE、Slack、Web 等开发环境结合；Codex 则可以在云环境中读取、编辑和运行代码，并后台并行执行任务。

Cognitive Workflow 与它们的差异在于：

| 对比项 | Claude Code / Codex | Cognitive Workflow |
|---|---|---|
| 核心领域 | 软件工程 / Coding | 跨领域复杂任务 |
| 主要能力 | 代码理解、修改、测试、PR | 任务规划、审查、修复、证据、交付 |
| 工作对象 | 代码库 | 文档、数据、报告、工程流程、科研流程、工具链 |
| 强项 | 代码级执行 | 复杂任务过程管理 |
| 局限 | 主要围绕软件工程 | 可扩展到科研、工程、项目资料、仿真、合规 |

一句话：

> **Claude Code / Codex 是代码工程 Agent，Cognitive Workflow 是复杂任务 Workflow Runtime。**

---

### 4. 与 OpenClaw / HermesAgent / Claude Cowork 的区别

OpenClaw 更偏运行在用户设备上的个人 AI 助理，可跨渠道响应用户并执行个人事务；HermesAgent 强调自我改进、持久记忆、从经验中创建技能并跨会话学习；Claude Cowork 则强调用户给 Claude 一个目标，它可以在本地文件和应用中自主工作并返回完成品。

Cognitive Workflow 与这些产品的区别在于：

| 对比项 | 个人助理型 Agent | Cognitive Workflow |
|---|---|---|
| 核心定位 | 长期个人助理 / 自主工作代理 | 复杂任务执行框架 |
| 记忆方式 | 用户偏好、历史交互、技能成长 | 项目记忆、节点产物、审查记录、失败修复历史 |
| 执行方式 | 自主处理任务 | 结构化 Workflow 执行 |
| 交付关注 | 完成用户委托 | 过程可审计 + 结果可交付 |
| 优势场景 | 日常事务、跨应用操作、个人自动化 | 长流程、高风险、需验证、需复现任务 |

一句话：

> **个人助理型 Agent 重在“替用户做事”，Cognitive Workflow 重在“把复杂任务做成可控工程”。**

---

### 5. Cognitive Workflow 的本质差异

Cognitive Workflow 和其他 Agent 的根本差异可以概括为六点。

#### 5.1 它不是 Chat，而是 Workflow

普通 Agent 以对话为主。Cognitive Workflow 以任务图为主，对话只是辅助输入和修改方式。

#### 5.2 它不是黑盒自主执行，而是可视化过程管理

用户可以看到节点、路径、输入、输出、审查规则和当前执行状态。

#### 5.3 它不是生成一次结果，而是追求可审查交付

每个关键结果都要经过评价节点判断。

#### 5.4 它不是失败后人工重来，而是进入修复闭环

不通过会触发失败诊断、修复节点、重新执行和再次审查。

#### 5.5 它不是依赖单一模型能力，而是通过 Workflow 补偿模型差异

通过节点契约、上下文包、证据包、结构化输出和多轮审查，把强模型的一次性综合能力拆成系统级多阶段流程。

#### 5.6 它不是只沉淀聊天记忆，而是沉淀可复用流程资产

成功 Workflow 可以变成模板，失败修复经验可以变成 Skill，项目上下文可以变成可复用 Harness。

---

### 6. 最终定位

Cognitive Workflow 的一句话定位是：

> **面向复杂长任务的认知型 Agent Workflow 工作台。**

更完整地说：

> **Cognitive Workflow 通过任务编排、节点契约、证据约束、独立审查、失败修复、人工检查点和状态记录，将复杂任务转化为可执行、可验证、可复现、可修复的 Agent 工作流，并尽可能提升不同模型条件下的交付一致性。**

如果说 Dify 解决的是：

```text
如何构建 AI 应用？
```

Manus 解决的是：

```text
如何让 Agent 自主完成任务？
```

Claude Code / Codex 解决的是：

```text
如何让 Agent 更好完成软件工程任务？
```

那么 Cognitive Workflow 解决的是：

```text
如何让 Agent 稳定完成复杂、长程、高风险、需审查、需复现的认知任务？
```

---

## 总结

Cognitive Workflow 的核心不是多一个 Workflow UI，也不是多一种 Agent 执行模式，而是提出了一种新的 Agent 工作范式：

> **用 Workflow 管住 AI，用审查校准 AI，用修复推进 AI，用记忆沉淀 AI。**

它的目标不是让 Agent 完全自主，也不是让用户手动控制每一步，而是在二者之间建立一种新的协作方式：

```text
Agent 负责执行和生成；
Workflow 负责结构和边界；
Review 负责质量判断；
Repair 负责失败修复；
Human 负责关键决策；
Memory 负责长期沉淀。
```

最终目标是：

> **把复杂任务从不可控的 Agent 对话，转化为可交付的认知工程流程。**
