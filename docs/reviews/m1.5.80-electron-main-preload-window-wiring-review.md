## Findings

- P1: the initial window wiring denied `window.open` but did not guard `will-navigate` or `will-redirect`; after privileged preload installation, a top-level navigation could expose `window.cw` to a non-CW page.
- Resolution: `installDesktopWindowNavigationGuard()` now registers both navigation events. Dev renderer source allows only the same `http://127.0.0.1:<port>` origin, packaged file source allows only the exact generated file URL, and dispose removes the listeners.
- P1: the initial app registered runtime IPC handlers but did not eagerly start the sidecar on app/window creation, leaving the `http_sse.md §7` main spawn sidecar / READY / renderer notify chain behind a lazy renderer IPC call.
- Resolution: `createCwDesktopElectronWindow()` now calls `runtime.ipc.startupHandlers.getStartupResult()` immediately after installing IPC/lifecycle wiring and broadcasts startup statuses to the window.
- P1: window recreation could leak the prior runtime session and double-register IPC handlers, especially on macOS close-all then activate flows.
- Resolution: `startCwDesktopElectronApp()` disposes any existing session before creating a new one; the window `closed` listener unregisters IPC handlers, lifecycle listeners, startup/shutdown window broadcasters, fanouts, and navigation guards, then clears the active session.
- P3: `dev` pointed at a real Electron launch path without a default development runtime command, so startup would fail unless `CW_RUNTIME_DEV_COMMAND` was manually set.
- Resolution: unpackaged Electron defaults to `uv run --package cw_runtime --extra runtime cw-runtime`; packaged apps continue to use resources runtime discovery, and `CW_RUNTIME_DEV_COMMAND` remains an explicit override.
