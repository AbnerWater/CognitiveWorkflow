## Initial Findings

The first review failed on exception-chain sanitization:

- `_encrypt_secret_value()` wrapped dirty encryptor exceptions with `raise ... from exc`; formatted traceback could expose `secret_ref` or plaintext token through the provider cause.
- SQLite/OSError write failures were also wrapped with `raise ... from exc`; formatted traceback could expose SQLite paths or provider/OS messages.

## Repair

- W1.4.65 write-path wrappers now raise sanitized `ProjectSecretStoreError` with `from None`.
- Regression coverage asserts `traceback.format_exception(...)` does not contain `secret_ref`, token material, or SQLite paths, and checks `__cause__ is None` plus `__suppress_context__ is True`.

## Verified

- The write API uses the accepted `runtime_harness.md` §6 table shape: `secret_id`, `alias`, `value_encrypted`, `scope`, `created_at`.
- `write_project_mcp_secret_material()` creates the secure DB/table as needed and upserts by `secret_id` without adding schema columns.
- Empty `ProjectMCPSecretMaterial` fails closed before DB creation.
- Round-trip, upsert, sanitized encrypt failure, and sanitized storage failure tests passed.
- No accepted spec, ADR, public schema, dependency, UI/API route, OAuth, macOS/Linux keychain, or real external provider scope drift was found.
