# M1.3.5 Node Runner Foundation Review

Date: 2026-06-17

## Scope

Reviewed W1.3.5 Runtime node runner foundation against:

- `specs/state_machines/workflow_run.md`
- `specs/schemas/stream_event.md`
- `specs/runtime_harness.md`
- `specs/schemas/evaluation_result.md`
- `specs/schemas/repair_patch.md`
- `specs/schemas/node_contract.md`
- `specs/protocols/agent_adapter.md`
- W1.3.4 run lifecycle memory and runbook boundary

This slice is deterministic runner infrastructure only. It does not implement real Adapter calls, LangGraph runtime orchestration, ContextBuilder/EvidenceBuilder, Human Decision resolution, SQLite/OTel persistence, or git snapshots.

## Initial Review Verdict

FAIL.

Findings:

- Repair overlay was attached to the repair attempt instead of the next target retry attempt.
- Failed evaluation default routing ignored `failure_type` and could treat a terminal fail route as a repair target.
- Generated `EvaluationResult.criterion_results` did not come from `EvaluationTaskNode.contract.criteria`.
- `human_checkpoint` entered `waiting_user` without appending a pending `decisions.jsonl` record.
- New StreamEvents used valid event types but omitted key spec payload fields.

## Fixes Applied

- Repair now registers a pending prompt overlay after `repair.patch_applied`; the next target execution retry writes `overlays/<target_attempt_id>.json` and attaches it to the target `ExecutionPack.effective_prompt_overlay` and `NodeAttempt.effective_prompt_overlay_ref`.
- Failed evaluation default routing now uses failure taxonomy within W1.3.5 supported capabilities and validates repair/human targets by node type.
- `EvaluationResult.criterion_results` and `criteria_hash` now derive from `EvaluationContract.criteria`.
- `human_checkpoint` writes a pending decision record before emitting `human.gate_required` and `run.paused`.
- Attempt, evaluation, repair, human, run completed, and run failed events include the key payload fields listed in `stream_event.md`.

## Follow-up Review Verdict

CONDITIONAL_PASS.

Conditions:

- Clarify that W1.3.5 does not execute `model_escalation` repair; `model_capability_limit` must not route to unsupported prompt repair.
- Update runbook wording so overlay timing matches implementation.

## Final Review Verdict

PASS.

Final fixes:

- `model_capability_limit` now defaults to `human_checkpoint` when a valid human target exists, or `abort` otherwise, until model-escalation repair exists.
- Runbook describes pending overlay registration and target retry attempt binding.
