## Findings

- P2: Initial loopback smoke bypassed React panel controls by calling `session.dispatch(...)` directly for open/search/select/mark-read/full-reload ack. That proved workbench command routing and SSE projection, but a broken React search `onChange` or button `onClick` handler could still pass.
- Resolution: the smoke now mounts `RuntimeWorkbenchShellReactView` with a fake DOM, opens the stream through the React action button, changes search through the search input's React `onChange`, and drives Select match / Mark read / Acknowledge through rendered buttons. The test session tracks pending dispatches so assertions wait for external-store updates inside React `act`.

## Confirmed Properties

- `open_runtime_stream_session` starts the active stream session so the shell can consume live SSE after opening.
- `acknowledge_full_reload` through the workbench command route resets `full_reload_required` and restarts the stream session.
- Loopback smoke uses real `fetch`, loopback HTTP, SSE framing, runtime connection info, auth/project/client headers, replay `Last-Event-ID`, and 412 `SE_SSE_REPLAY_NOT_FOUND`.
- Live stream events pass through store, view model, interaction, workbench host, presenter, adapter, and shell snapshot before assertions.
- React shell controls for open stream, search, selected match, mark-read, and full-reload acknowledgement are covered against the live shell snapshot.
