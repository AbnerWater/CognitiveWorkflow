## Initial Blocking Finding

`Plato` reported a READY-transition race: the pre-READY `exit` listener was removed when READY was parsed, while orchestration installed the post-READY listener only after `startRuntimeSidecar()` resolved. If a sidecar emitted `READY <port>` and exited in the same tick, no listener observed the exit, leaving the registry entry and `runtime.lock` stale.

## Fix

- `RuntimeSidecarSession.closed` is now created before READY waiting and remains armed across the READY transition.
- `startRuntimeOrchestration()` binds ownership cleanup to `sidecar.closed`.
- Cleanup remains idempotent and uses matching connection unregister, so an older sidecar exit cannot remove a newer registry entry.
- Regression tests cover READY-after-exit cleanup, READY-transition same-tick exit, explicit stop, and newer registry protection.

## Re-Review Result

`Dalton` reported PASS with no blocking findings. The re-review confirmed the READY-transition race is covered by the current fix and by the regression test that emits READY and `exit` in the same transition.
