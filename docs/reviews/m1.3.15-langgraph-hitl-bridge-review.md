## Initial Findings

### High: stale LangGraph schedule mutated CW state before rejection

`LangGraphRuntimeNodeExecutor` called `advance_workflow_run()` before proving that the LangGraph node matched CW `run.json.current_node_ids`. A stale LangGraph state/checkpoint could therefore append jsonl or update `run.json` for the CW current node, then raise after the mutation.

Fix: every bridge call now merges the LangGraph node id into `NodeAdvanceRequest.node_id` before invoking the runner. The runner rejects mismatched nodes before persistence via the existing node-state forbidden transition path. A regression test asserts stale schedule rejection leaves `run.json` and StreamEvent ids unchanged.

### Medium: wrong error code for multiple active nodes

The initial bridge used `WR_CONCURRENT_RUN_FORBIDDEN` for `len(current_node_ids) > 1`. That code is reserved for overlapping nonterminal WorkflowRuns, not multiple active nodes inside one run.

Fix: the bridge now raises `WR_STATE_FORBIDDEN_TRANSITION` for unsupported multiple current nodes. A regression test locks this behavior.
