## Initial Review

Result: FAIL

Blocking finding:

- P1: stale cleanup and release used inspect/read followed by path-based unlink, so another process could replace `runtime.lock` between the check and delete. That could delete a fresh lock and violate the runtime harness invariant that only one sidecar may start under `locks/runtime.lock`.

## Repair

- Added a short-lived `runtime.lock.mutation` guard used by both stale cleanup and release.
- Stale cleanup now re-inspects `runtime.lock` while holding the guard before unlinking.
- Release now holds the same guard and rechecks that the current file content still matches the lease content before unlinking.
- Added a regression test for the stale-cleanup interleaving where a lock is replaced before guarded cleanup.
- Updated the runbook to state that this is still an exclusive-create + mutation guard skeleton, not the final `flock` / `LockFileEx` implementation.
