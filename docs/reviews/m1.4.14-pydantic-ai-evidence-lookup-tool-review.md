# M1.4.14 PydanticAIAdapter Evidence Lookup Tool Review

## Scope

- Review target: `apps/runtime/src/cw_runtime/adapters/pydantic_ai_adapter.py`
- Tests: `apps/runtime/tests/test_m1_4_1_pydantic_ai_adapter.py`
- Progress log: `docs/04_runbook/m1.4-progress.md`

W1.4.14 adds adapter-owned `evidence_lookup` injection for the default Pydantic AI SDK session. The review checked that this stays limited to `ExecutionPack.evidence_pack`, does not route the builtin through the external toolset factory, and does not claim unrelated builtin tools or lifecycle work.

## First Review

Result: FAIL

- P1: missing evidence IDs were only proven by direct callable invocation. The SDK stream/run path did not prove a structured failed attempt carrying `EP_RUNTIME_EVIDENCE_LOOKUP_MISS`.
- P1: default capabilities over-advertised generic `tool_call` and `max_tool_iterations` without a configured toolset factory.

## Fixes

- `_iterate_session()` now maps evidence lookup miss exceptions to a terminal `attempt.failed` with `AA_RUN_TOOL_NOT_FOUND` and payload `runtime_error_code=EP_RUNTIME_EVIDENCE_LOOKUP_MISS`.
- The default capability descriptor keeps generic `tool_call=False` and `max_tool_iterations=0` unless a `PydanticAIToolsetFactory` is configured, while still declaring `evidence_lookup_tool=True`.
- Tests now cover both direct `evidence_lookup("ev_fact")` success and SDK stream exception mapping for missing evidence IDs.
- Runbook wording now states that generic tool routing remains factory-gated and does not claim `web_fetch`, `file_io`, `python_sandbox`, SkillRegistry, MCP lifecycle, budget ledger, or cancel completion.

## Final Review

Result: PASS

- Independent subagent confirmed the two first-review P1 findings are fixed.
- No remaining blockers were reported.
