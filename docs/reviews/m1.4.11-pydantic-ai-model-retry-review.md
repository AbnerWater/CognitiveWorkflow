## Review Scope

Reviewed the W1.4.11 narrow slice against `specs/protocols/agent_adapter.md`, `specs/schemas/node_contract.md`, `specs/schemas/stream_event.md`, `docs/04_runbook/m1.4-progress.md`, and repository `AGENTS.md`.

Focus areas:

- `retry_policy.model_retries` is not silently dropped or incorrectly projected into SDK `AgentRetries`.
- Retry happens only for retryable model/session exceptions before any raw stream event is emitted.
- Usage-limit, adapter-coded errors, output validation, cancel/timeout, toolset failures, and partial-stream failures are not retried.
- Retry exhaustion uses a spec-coded adapter error with useful payload.
- Capabilities remain truthful; no new dependency or forbidden direct `pydantic_ai` import.

## Findings

Initial review failed because retry classification was too broad:

- HTTP status exceptions such as `ModelHTTPError` could be retried without checking status code, so non-transient `4xx` failures could be replayed.
- Toolset factory/setup exceptions raised inside SDK session startup could be misclassified as model retry candidates.

Both findings were fixed and re-reviewed.

## Evidence

- `run()` retries only `AA_RUN_STREAM_INTERRUPTED` errors marked retryable and only before any raw event is received; failed sessions are discarded before replay.
- HTTP status retries are limited to `408`, `429`, and `>=500`; non-retryable HTTP status failures still surface as `AA_RUN_STREAM_INTERRUPTED` with `retryable=false`.
- Toolset factory exceptions are wrapped as non-retryable `AA_RUN_TOOL_NOT_FOUND`.
- SDK retries still project only `tools` and `output`; `model_retries` is adapter-owned.
- Tests cover success retry, exhaustion, HTTP `400` no-retry, HTTP `500` retry, partial-stream no-replay, toolset factory no-retry, per-tool retry fail-closed, and usage-limit mapping.
