# M1.3.9 ModelRouter Phase 1 Review

## Scope

W1.3.9 adds the static Phase 1 ModelRouter foundation:

- `cw_runtime.model_router` defines static model profiles, adapter descriptors, routing requests, routing decisions, routing traces, and deterministic `route_model()`.
- Project model settings are loaded from `.agent-workflow/settings.json.models` and merged with optional global profiles plus built-in fallback profiles.
- The runner routes before `ExecutionPack` creation, writes `runs/<run_id>/routing.jsonl`, and copies `adapter_id`, `model_profile_id`, and `effective_model_settings` into attempt artifacts.
- `routing.jsonl` is initialized at run creation.

## Independent Review

Initial verdict: `FAIL`.

Findings:

- Final `RoutingDecision.escalation_chain` allowed local-to-cloud adjacency until an actual escalation trigger, which could persist an invalid chain.
- `primary_model_profile_id="auto"` ignored `WorkflowModelPolicy.default_model_profile_id` and project defaults during tie-break.
- Sensitive cloud primary/filter paths did not surface `MR_SENSITIVE_DATA_REMOTE_FORBIDDEN`.
- Direct public static builder usage could emit empty `effective_model_settings` after runner integration moved settings injection into ModelRouter.
- `CLAUDE.md` remained dirty and must stay out of the W1.3.9 commit.

Fixes:

- Final escalation-chain validation now enforces D-MR-4 before `RoutingDecision` is returned.
- Auto routing ranks workflow/project defaults before normal score tie-break.
- Sensitive cloud primary/filter paths raise `MR_SENSITIVE_DATA_REMOTE_FORBIDDEN`.
- Static builder direct usage falls back to contract `model_settings`; regression coverage was added.
- Deterministic runner test projects explicitly clear project-level cloud escalation chains to avoid spec-invalid local-to-cloud decisions.

Follow-up verdict: `CONDITIONAL_PASS`.

Condition: keep the unrelated dirty `CLAUDE.md` out of the W1.3.9 commit.
