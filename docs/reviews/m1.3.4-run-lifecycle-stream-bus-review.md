# M1.3.4 Run Lifecycle Stream Bus Review

Date: 2026-06-17
Scope: W1.3.4 `cw_runtime.runs` WorkflowRun lifecycle, StreamEvent jsonl persistence, and runtime SSE endpoints.

## Independent Review

Initial verdict: `FAIL`

Findings:

- Run pause/resume routes were registered under `/runs/{run_id}/pause|resume`, while `http_sse.md` defines pause/resume as workflow-level actions and only keeps run-level `cancel`.
- `/runs/{run_id}/stream` returned a finite replay-only response instead of a live SSE stream with heartbeat/replay semantics.
- Active-run protection checked for existing non-terminal runs before writing without holding `runtime.lock`.
- Generic resume allowed `waiting_user` and `repairing` to move to `running`, bypassing the required `decision.resolved` and `patch.applied` triggers.
- `runs/<run_id>/stream-events/` was not protected from Git tracking.
- Persisted `run.json` included a spec-external `schema_version` field.

## Resolution

- API routes now use workflow-level `POST /cw/v1/workflows/{workflow_id}/pause|resume|cancel`; run-level `POST /cw/v1/runs/{run_id}/cancel` remains.
- Run streams now use `StreamingResponse` backed by `stream_sse_events()`, with `Last-Event-ID` preflight, persisted replay, filter support, polling, and `system.heartbeat`.
- Run creation and run/event mutations are serialized under `runtime.lock`; concurrent start requests are covered by regression tests.
- Generic resume only allows `paused -> running`; `waiting_user` and `repairing` remain reserved for later decision/repair flows.
- Project initialization now ignores and pre-commit-blocks run and planning stream-events.
- `WorkflowRunDocument` no longer persists `schema_version` into `run.json`.
