## Evidence

- Boundary is respected: the production code change is limited to the in-memory desktop runtime connection registry, uses only Node stdlib `fs.realpathSync.native`, and does not add dependencies, Electron imports, IPC registration, `BrowserWindow` wiring, renderer code, SSE/EventSource behavior, token/port persistence, or `runtime.lock` schema changes.
- Display identity is preserved: the stored `projectRoot` remains the caller path normalized with `path.resolve()`, while realpath is used only for the internal lookup key.
- Canonicalization is shared consistently across `register()`, `get()`, the handoff `resolver()`, and `unregister()`, so symlink/junction aliases and target paths can match the same in-memory connection.
- Fallback behavior is fail-soft: resolver failures and `null` / `undefined` realpath results fall back to the resolved display path, while `projectRootRealpath: false` explicitly disables realpath lookup.
- Existing `projectRootCaseSensitivity` behavior still applies after canonicalization, keeping Windows default lookup case-insensitive and other platforms case-sensitive unless configured otherwise.
- Tests cover symlink/junction alias matching through an injected resolver, unavailable realpath fallback, explicit realpath disable, empty realpath rejection, and existing case-sensitive / case-insensitive behavior.
