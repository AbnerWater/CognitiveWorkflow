## Scope

Reviewed the W1.4.4 narrow slice against `specs/protocols/agent_adapter.md`, ADR-0002, and `docs/04_runbook/m1.4-progress.md`.

The review focused on:

- Whether `ExecutionPack.usage_limits` is projected into the Pydantic AI SDK session request.
- Whether both `agent.run_stream_events(...)` and fallback `agent.run(...)` pass SDK `UsageLimits`.
- Whether CW token fields map to Pydantic AI 1.107.0 usage-limit fields without introducing request-count limit semantics.
- Whether unsupported `max_cost_usd` is rejected rather than silently ignored.
- Whether SDK `UsageLimitExceeded` becomes structured `AA_RUN_USAGE_LIMIT`.
- Whether the change avoids scope creep into Toolset/MCP, HITL/deferred resume, cancel, UI, cost accounting, multimodal input, or evidence lookup.

## Findings

Verdict: PASS.

The reviewers confirmed:

- Token limits are projected from `ExecutionPack.usage_limits` into `PydanticAIRunRequest`.
- `usage_limits` is passed to both SDK streaming and fallback run paths.
- `max_input_tokens`, `max_output_tokens`, and `max_total_tokens` map to `input_tokens_limit`, `output_tokens_limit`, and `total_tokens_limit`.
- `request_limit=None` is intentional because Pydantic AI 1.107.0 defaults `request_limit` to `50`, while CW W1.4.4 does not define request-count limiting.
- `max_cost_usd` is explicitly rejected with `AA_PREPARE_INCOMPATIBLE_ADAPTER`.
- SDK `UsageLimitExceeded` is translated to `AA_RUN_USAGE_LIMIT`.
- Capabilities remain conservative and do not advertise Toolset/MCP/HITL/deferred/cancel/multimodal/evidence features.
- Tests cover stream token mapping, fallback token mapping, cost rejection, usage-limit error mapping, and capabilities.
