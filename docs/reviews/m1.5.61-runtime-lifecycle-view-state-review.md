## Findings

Initial review found one P2 sensitive/raw-content boundary issue: `RuntimeLifecycleViewStateItem` exposed `reason?: string`, startup/shutdown mapping copied upstream `reason`, and `shutdown_failed` summary concatenated the free-text reason. That could allow future error text, URLs, tokens, prompt text, or model output snippets to pass into shell timeline state.

Resolution: `RuntimeLifecycleViewStateItem` no longer exposes `reason`. Startup item summaries are now fixed by status kind instead of reusing upstream view-model summaries. Shutdown item summaries do not include reason, and `shutdown_failed` uses the fixed summary `Runtime shutdown failed.`. `contract.test.ts` adds a sentinel test with `token_abc`, `base_url`, `prompt`, and `model output` in upstream startup/shutdown reasons and asserts the view state does not expose those strings or a `reason` field.

Follow-up review passed. No blocker, P1, or P2 findings remain.

## Confirmed Alignment

- The implementation stays renderer-side pure TypeScript.
- No React or Electron import, production dependency, package or lockfile change, accepted spec / ADR change, schema change, generated-code content change, or new IPC channel was introduced.
- The adapter consumes the W1.5.60 lifecycle status controller snapshot without changing controller contracts.
- Readiness and primary action projection is stable: idle/stopped to `start_runtime`, starting/waiting/shutting_down to `wait`, ready to `none`, and blocked/timed_out/failed to `retry_startup` or `inspect_issue`.
- Timeline items merge startup view-model items and shutdown status items while exposing only safe renderer-shell fields.
- Listener snapshots are isolated, listener / `onError` exceptions are isolated, and dispose releases the controller subscription.
