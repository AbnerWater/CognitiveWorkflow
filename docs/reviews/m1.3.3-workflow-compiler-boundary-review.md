# M1.3.3 Workflow Compiler Boundary Review

Date: 2026-06-17
Scope: W1.3.3 `cw_runtime.engine` WorkflowGraph validation and Engine IR compile boundary.

## Independent Review

Initial verdict: `FAIL`

Findings:

- The W1.3.3 implementation produced `EngineWorkflowIR`, while `workflow_graph.md §7` and ADR-0004 still require the full M1.3 compiler to produce LangGraph `StateGraph`.
- Runtime validation collapsed some `cw_schemas` custom errors, including `NC_*` and `WG_L2_METADATA_NOT_NAMESPACED`, into `WG_L1_INVALID_JSON`.
- Human checkpoint route merging could drop distinct decision keys when multiple decisions target the same node.
- L4 model registry checks included non-LLM node contract policies.
- Repair route merging treated `repair` edges as alternatives to post-repair `retry` edges.

## Resolution

- W1.3.3 is documented as Graph -> Engine IR / LangGraph adapter seam only; the final WorkflowGraph -> LangGraph `StateGraph` compiler remains a later M1.3 completion condition.
- `WorkflowValidationError` now derives levels for `WG_*` and `NC_*` spec custom error codes and preserves those codes from Pydantic validation.
- Human decision routing now preserves one route per decision key, including when an explicit `human` edge already exists.
- L4 model registry checks only inspect node-level model policies for LLM-backed nodes: `execution_task`, `evaluation_task`, and `repair_task`.
- Repair node route merging now generates/reuses only `retry` route semantics for post-repair flow.
