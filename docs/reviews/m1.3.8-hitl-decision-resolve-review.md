# M1.3.8 HITL Decision Resolve Review

## Scope

W1.3.8 adds the first runtime-owned Human Checkpoint resolution path:

- `resolve_human_decision()` accepts an explicit `HumanDecisionRequest` only while the run is `waiting_user`.
- Resolved decisions are appended to `decisions.jsonl` and preserve the pending record's `requested_at`.
- The runner emits `human.gate_resolved`, advances the human node state, emits `run.resumed`, and routes by compiled `human` edges keyed by decision.
- `POST /cw/v1/runs/{run_id}/decisions` exposes the same flow over the local sidecar API with auth, `ErrorEnvelope`, and idempotency replay.

## Independent Review

Initial verdict: `FAIL`.

Findings:

- Missing `schema_version` was accepted by `POST /cw/v1/runs/{run_id}/decisions` because `HumanDecisionRequest` had a default.
- Successful decision responses omitted the HTTP body `schema_version` required by `http_sse.md §11`.
- `reject/edit` Node-level semantics are deferred: W1.3.8 resumes the run by declared decision route and does not yet implement `reject -> failed` or `edit -> retrying` resumption.

Fixes:

- `HumanDecisionRequest.schema_version` is now required.
- Decision endpoint success responses now include `schema_version`.
- Endpoint tests cover missing `schema_version`, unsupported `schema_version`, and success response versioning.

Recorded boundary:

- W1.3.8 only closes Run-level `decision.resolved -> running` plus declared decision route. Node-level `reject/edit` terminal/retry semantics remain a later HITL/Adapter resumption item.

Follow-up verdict: `PASS`.

No remaining required fixes. The reviewer confirmed the two P1 API contract issues are closed and accepted the documented `reject/edit` limitation as W1.3.8 residual risk.
