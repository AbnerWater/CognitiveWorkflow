## Initial Finding

Hegel (`019ef7ae-4da8-7a02-b602-716051a02054`) initially returned FAIL with one P2 finding:

- The first implementation caused `createRuntimeStreamEventStore()` to prefer `client.subscribeAll(listener)` whenever available, so production real-session paths ignored configured `eventTypes` and would record known-but-unconfigured events such as `tool.call_started`.

## Fix

- `CreateRuntimeStreamEventStoreOptions` now accepts optional `knownEventTypes`.
- `createRuntimeStreamInteractionSession()` passes `RUNTIME_STREAM_ALL_EVENT_TYPES` into the store.
- Store all-event subscription now records only:
  - configured event types from `eventTypes`; or
  - event types outside `knownEventTypes`, which are forward-compatible unknown events.
- Known-but-unconfigured event types are filtered out.
- Store falls back to exact subscriptions when `knownEventTypes` is absent or the client lacks `subscribeAll()`.
- The real session test now emits an unconfigured known `tool.call_started`, configured `model.text_delta`, and unknown `adapter.experimental_event`, asserting only the configured and unknown events enter the view model.

## Final Review Result

Hegel re-reviewed the updated diff and returned PASS with no remaining findings.

Confirmed:

- Unknown SSE event names flow through the internal all-event channel into store/session/view model.
- Exact `"error"` transport/replay events remain excluded from all-event fanout and are not recorded as StreamEvent payloads.
- Exact event subscriptions remain valid for legacy/fake clients and direct store use without `knownEventTypes`.
- Diff is limited to desktop runtime stream client/store/session/fetch source tests and docs.
- No spec/schema/generated/dependency/runtime API/IPC/`window.cw`/storage changes were introduced.
