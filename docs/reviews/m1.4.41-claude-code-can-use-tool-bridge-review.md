## 1. Scope

Reviewed the W1.4.41 change against `specs/protocols/agent_adapter.md` §9.1, `specs/schemas/node_contract.md` MCP `requires_approval` semantics, `specs/schemas/stream_event.md` approval/HITL event ordering, ADR-0009, and `docs/04_runbook/m1.4-progress.md`.

Focus areas:

- default W1.4.19 `permission_mode="dontAsk"` behavior when the bridge is disabled
- opt-in `can_use_tool` and `PreToolUse` hook projection
- MCP `requires_approval` handling, including `mcp__<server>__*` wildcard overlap
- `tool.approval_required -> human.gate_required -> human.gate_resolved -> tool.approved/rejected` event ordering
- fake SDK coverage for approve, reject, and unpermitted-deny paths

## 2. Review Findings

Initial review verdict: FAIL.

- P1: approval-required MCP tools could still be auto-approved through an SDK `mcp__<server>__*` wildcard when the same pack also carried a specific `MCPToolRef(..., requires_approval=true)`.
- P2: the resume path emitted `human.gate_resolved` but did not emit the required `tool.approved` / `tool.rejected` StreamEvent.

Repairs:

- `_claude_auto_approved_tools()` now subtracts approval-required refs by overlap, so a wildcard that covers an approval-required tool is removed from SDK `allowed_tools`.
- `ClaudeCodeAdapter.resume()` now emits `tool.approved` or `tool.rejected` after `human.gate_resolved` and before continuing the SDK/session.
- Tests now cover bridge approve, bridge reject, default no-callback `dontAsk`, unpermitted deny, wildcard subtraction, and the legacy injected `permission_prompt` path.

Final review verdict: PASS. No remaining actionable findings were reported.
