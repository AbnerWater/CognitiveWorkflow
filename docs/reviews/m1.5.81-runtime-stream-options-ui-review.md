## Findings

- P1: The first fetch-backed SSE source silently returned on clean body EOF. The existing reconnect controller only enters replay/reconnect through the EventSource-compatible `error` listener, so a dropped response could leave the UI in `running` with no live connection and no `Last-Event-ID` replay.
- Fix: `readRuntimeFetchEventSourceBody()` now emits an `error` event after non-user-initiated EOF. The emitted payload is `{ "reason": "Runtime stream connection closed" }`, which the existing reconnect failure normalizer can consume. Explicit `close()` still aborts and suppresses further events.
- Regression coverage: `runtime-stream-fetch-event-source.test.ts` now closes the response body after a heartbeat and asserts one parsed event plus one EOF error event.

## Confirmed Properties

- Runtime stream UI builds only existing `CreateRuntimeStreamInteractionSessionFactorySessionOptions`.
- Run streams use `/runs/{run_id}/stream` and planning streams use `/workflow-planning/sessions/{session_id}/stream` through the existing stream client path builder.
- Run channel categories exclude `planning`; planning channel categories are limited to `planning` and `system`.
- Stream ids reject empty values, path separators, query/hash characters, `..`, whitespace, and control characters before dispatch.
- Seq filters are optional non-negative safe integers, and `untilSeq < sinceSeq` disables stream opening.
- Fetch-backed SSE source forwards existing request headers, parses SSE frames, maps replay failures and clean EOF into the existing error event path, and aborts fetch on close.
- The renderer entry injects the fetch-backed factory instead of the previous unavailable placeholder.
