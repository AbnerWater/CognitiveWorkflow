## W1.4.1 范围

本切片启动 M1.4 AgentAdapter 工作，先关闭 `agent_adapter.md` 中已经声明的 `pydantic_ai` entry point 指向缺失模块的问题，并建立可测试的 PydanticAIAdapter foundation；不新增 production dependency、不要求默认环境安装 `pydantic-ai-slim`、不实现真实 SDK Agent/Toolset 构造：

- 新增 `cw_runtime.adapters.pydantic_ai_adapter`，导出 `PydanticAIAdapter`、`PydanticAIRunRequest`、`PydanticAIResumeRequest`、`PydanticAISession`、`PydanticAISessionFactory`、`RawPydanticAIEvent` 与 `build_pydantic_ai_descriptor()`。
- `PydanticAIAdapter.capabilities()` 引入 PydanticAI 能力声明 scaffold；W1.4.2 已按当前真实实现收敛路由面能力，避免 ModelRouter 选择未完成的 Toolset/HITL/cancel 路径。
- `prepare()` 只消费 `ExecutionPack`，生成 `AttemptHandle(state=prepared)`，并缓存 pack；不通过 side channel 注入参数。
- `run()` 经注入的 `PydanticAISession` seam 转译 raw events 到 spec StreamEvent：`model.text_delta`、`model.thinking_delta`、`tool.call_started`、`tool.call_completed`、`tool.approval_required`、`human.gate_required`、`model.request_completed`、`attempt.completed`、`attempt.failed`；若 session iterator 耗尽但未产出 terminal raw event，则兜底产出 `AA_RUN_INTERNAL` 的 `attempt.failed`。
- `resume()` 支持 `human_decision` 与 `deferred_tool` 两类 resumption；human decision 先发 `human.gate_resolved`，deferred tool result 不额外发 human event。
- `cancel()` 标记 handle cancelled 并委托 session cancel；`finalize()` 返回 `AttemptOutcome`，包含 output hash、AdapterError、AttemptProvenance 与 Pydantic AI traceparent 占位。
- 输出 schema 在 foundation 层覆盖 object required/type 校验，失败映射 `AA_RUN_OUTPUT_VALIDATION_FAILED`。

边界说明：本切片只建立 adapter 协议与事件转译闭环，不直接 import `pydantic_ai` SDK。当前默认 `.venv` 中 `cw-runtime.exe` 被已有进程占用，`uv run --package cw_runtime --extra agents ...` 无法完成环境切换；真实 SDK-backed `Agent.iter()`、Toolset/MCP/ApprovalRequiredToolset 构造、usage/messages 回填、≤5s cancel 实测留给后续 M1.4 切片。

## W1.4.2 范围

本切片把 `PydanticAIAdapter(session_factory=None)` 从 foundation-only 报错路径推进为默认 SDK-backed session 入口；仍保持 optional dependency lazy-load，不让默认 runtime import 依赖 `pydantic_ai`：

- 新增 `PydanticAISDKSession`，仅在实际运行默认 session 时 `importlib.import_module("pydantic_ai")`；缺少 optional `agents` extra 时仍转为结构化 `AA_RUN_INTERNAL`。
- 默认 SDK session 使用 Pydantic AI 1.107.0 的最小稳定接口：`Agent(model, output_type=StructuredDict(schema), system_prompt=..., instructions=..., model_settings=..., defer_model_check=True)` 与 `await agent.run(user_prompt, model_settings=...)`。
- 将 `ExecutionPack.node_contract_snapshot.output_schema` 映射为 `StructuredDict`；空 schema 时使用 permissive object schema，保持 CW Adapter output 为 object。
- 将 SDK `AgentRunResult.output` 转为 `completed` raw event，并把 usage/messages/traceparent 回填到 `AttemptOutcome.usage`、`AttemptOutcome.messages` 与 `AttemptProvenance.pydantic_ai_traceparent`。
- usage 回填显式映射 Pydantic AI `cache_write_tokens/cache_read_tokens` 到 CW `cache_creation_input_tokens/cache_read_input_tokens`，避免外部 SDK extra 字段进入 `RunUsage(extra=forbid)`。
- `capabilities()` 当前仅声明已由 W1.4.2 默认 SDK session 实现的路由面能力：`chat`、structured output、cloud/private/local provider、model settings passthrough；`streaming/tool_call/mcp/human_in_the_loop/deferred_tool_results/multi_modal/cancel/evidence_lookup_tool` 先保持 false/empty，后续切片实现真实语义后再打开。

边界说明：W1.4.2 不实现 `Agent.iter()` 细粒度事件流、Toolset/MCP/ApprovalRequiredToolset 构造、human decision resume、deferred tool result SDK resume、usage limit、真实 API key 调用或 cancel ≤5s 实测；这些仍留在后续 M1.4 切片。

## W1.4.3 范围

本切片把 W1.4.2 的默认 SDK-backed session 从一次性 `agent.run(...)` 推进到 Pydantic AI 1.107.0 的 `agent.run_stream_events(...)` 事件流桥接；仍保持 optional dependency lazy-load：

- `PydanticAISDKSession.run()` 优先使用 `async with agent.run_stream_events(...) as stream`，按 SDK 推荐生命周期消费事件流；缺少该接口时回退到 W1.4.2 的 `agent.run(...)` 路径。
- 新增 SDK 事件对象到现有 raw event seam 的转换：`PartStartEvent/PartDeltaEvent/PartEndEvent` 的 text/thinking delta 与 completed、`FunctionToolCallEvent`、`FunctionToolResultEvent`、`BuiltinToolCallEvent/BuiltinToolResultEvent`、`AgentRunResultEvent`。
- raw event 到 StreamEvent 的 adapter 层补齐 `model.text_completed`、`model.thought_completed`、`tool.call_failed`，继续复用 `StreamEvent` spec 内类型，不新增事件名。
- `capabilities().streaming=True` 现在由默认 SDK session 的真实 stream bridge 支撑；`tool_call/mcp/human_in_the_loop/deferred_tool_results/multi_modal/cancel/evidence_lookup_tool` 仍保持 false/empty，因为本切片未构造 Toolset/MCP/ApprovalRequiredToolset，也未实现 SDK resume/cancel 实测。
- 保留注入式 `PydanticAISession` raw-event seam，便于后续 M1.4 子阶段分别接入 Toolset/MCP/HITL/deferred resume 与 cancel。

边界说明：W1.4.3 不实现 `Agent.iter()` run graph 控制、Toolset/MCP 构造、ApprovalRequiredToolset、human decision SDK resume、deferred tool result SDK resume、usage limit、真实 API key 调用或 cancel ≤5s 实测；这些仍留在后续 M1.4 切片。

## W1.4.4 范围

本切片关闭 `ExecutionPack.usage_limits` 在默认 Pydantic AI SDK session 中被忽略的问题；范围只覆盖 SDK 可原生执行的 token usage limits，不打开 Toolset/MCP/HITL/deferred/cancel 等能力：

- `PydanticAIRunRequest` 带上 `usage_limits` 投影，来自 `ExecutionPack.usage_limits.model_dump(exclude_none=True)`。
- 默认 SDK session 在 `agent.run_stream_events(...)` 与 fallback `agent.run(...)` 两条路径都传入 Pydantic AI `UsageLimits`。
- CW 字段映射到 Pydantic AI 1.107.0 字段：`max_input_tokens -> input_tokens_limit`、`max_output_tokens -> output_tokens_limit`、`max_total_tokens -> total_tokens_limit`，并设置 `request_limit=None`，避免引入 CW spec 外的 request-count 限流语义。
- SDK 抛出的 `UsageLimitExceeded` 转译为结构化 `AA_RUN_USAGE_LIMIT` attempt failure，不再落入 `AA_RUN_INTERNAL`。
- `max_cost_usd` 当前不能由 Pydantic AI SDK 原生执行；本切片显式以 `AA_PREPARE_INCOMPATIBLE_ADAPTER` 拒绝，而不是静默忽略。

边界说明：W1.4.4 不实现成本统计/预算扣减、usage-limit UI、Toolset/MCP、ApprovalRequiredToolset、human/deferred SDK resume、真实 cancel ≤5s、multimodal input 或 evidence lookup tool。

## W1.4.6 范围

本切片关闭 Pydantic AI 默认 SDK session 对 `ExecutionPack.retry_policy` 中 SDK 可直接承载维度的忽略问题；范围只覆盖 Pydantic AI 1.107.0 `Agent(retries=...)` 的真实能力：

- `PydanticAIRunRequest` 新增 `PydanticAIRetryPolicy` 投影，当前用来承载 SDK 可映射字段以及不支持项的 fail-closed 检查。
- 默认 SDK session 将 `output_validation_retries -> retries["output"]`，整数型 `tool_retries -> retries["tools"]` 传入 `Agent(...)`。
- `tool_retries` 为 per-tool map 时，当前 SDK `AgentRetries` 无法直接表达；本切片以 `AA_PREPARE_INCOMPATIBLE_ADAPTER` fail closed，不静默降级成全局工具重试。
- Pydantic AI 1.107.0 的 `AgentRetries` 只有 `tools/output`，没有 spec 中早期设想的 `model` 字段；W1.4.6 先对显式非零 `model_retries` fail closed，避免无法由 SDK 执行却静默丢弃。W1.4.11 已把该字段迁移到 adapter-owned retry，不再传给 SDK。

边界说明：W1.4.6 不实现 `retry_policy.max_attempts`（NodeRunner 已负责 attempt 级上限）、`backoff/escalation_after`（ModelRouter/runner 责任）、`timeout_seconds`（后续 cancel/timeout 切片处理）、per-tool retry map、成本预算、human/deferred SDK resume 或真实 registry。

## W1.4.7 范围

本切片关闭 Pydantic AI 默认 SDK session 对 human/deferred resume 的缺口；范围只覆盖 Pydantic AI 1.107.0 `DeferredToolRequests` / `DeferredToolResults` 的真实 API，不实现真实 registry、cancel 时限或成本预算：

- 当 `ExecutionPack` 声明 toolset 且配置了 `PydanticAIToolsetFactory` 时，默认 SDK session 将 `output_type` 扩展为 `[StructuredDict(...), DeferredToolRequests]`，允许 SDK 以 deferred control-flow output 结束。

- SDK `DeferredToolRequests.approvals` 转译为 `tool.approval_required` + `human.gate_required`，支持 `human_decision` 生成 approval result。

## W1.4.8 范围

本切片关闭 `ExecutionPack.retry_policy.timeout_seconds` 在 PydanticAIAdapter 默认/注入 session 路径中被忽略的问题；范围只覆盖 `node_contract.md` §11 明确要求的 adapter 层 attempt 总时限控制：

边界说明：W1.4.8 不实现真实 ToolRegistry / SkillRegistry / MCP registry，不实现用户主动 `cancel()` ≤5s 实测，不接入 model HTTP retry utilities，不实现成本预算/扣减，也不推进 ClaudeCodeAdapter 真实 SDK/session。

## W1.4.9 范围

本切片关闭 PydanticAIAdapter 主动 `cancel()` 只能设置 handle 状态、但无法唤醒正在等待 session 下一帧的 `run()` / `resume()` iterator 的问题；范围只覆盖 adapter seam 内的 iterator 退出控制，不宣称真实 SDK 上游推理已被实测终止：

- 为每个 prepared handle 建立 cancel event，`cancel()` 设置 `cancellation_requested`、记录 `CancelReason`，并唤醒正在等待 session event 的 adapter iterator。
- `run()` / `resume()` 的 session event 消费同时等待下一帧、cancel event 与 `retry_policy.timeout_seconds` deadline；主动 cancel 会取消当前 `__anext__()` future，并以既有 `AA_RUN_CANCELLED` 产出终端 `attempt.failed`。
- `resume()` 路径同样支持主动取消，确保 `human.gate_resolved` 后阻塞在 session resume 时也能退出。
- `capabilities().cancel` 继续保持 `False`，因为当前默认环境未安装 optional `pydantic_ai`，本切片未对真实 SDK-backed upstream cancellation 做端到端实测。

边界说明：W1.4.9 不实现真实 ToolRegistry / SkillRegistry / MCP registry，不接入 model HTTP retry utilities，不实现真实 SDK 上游 `cancel()` ≤5s 实测，不实现成本预算/扣减，也不推进 ClaudeCodeAdapter 真实 SDK/session。

## W1.4.10 范围

本切片把 W1.4.5 的 `PydanticAIToolsetFactory` seam 收敛成可复用的 registry-backed resolver；范围只覆盖"已经由运行时上游构造好的 SDK toolset"如何按 `ExecutionPack` 声明解析到 Pydantic AI `Agent(toolsets=...)`：

- 新增 `PydanticAIToolsetRegistry`，支持注册 builtin toolset、`skill_id@version` resolved skill toolset 与 MCP server toolset，并通过 `as_factory()` 复用既有 SDK session seam。
- registry 按 `PydanticAIToolsetRequest` 的声明顺序解析 builtin、resolved skill、MCP server，去重后保持确定性输出。
- 任一声明的 builtin / skill / MCP server 未注册时，以 `AA_RUN_TOOL_NOT_FOUND` fail closed，并在 error payload 中列出缺失项；不会退化成裸模型运行。
- MCP server toolset 仍经过既有 `ApprovalRequiredToolset` 包装逻辑，保持 server 作用域和 `tool_name` / `*` approval predicate。

## W1.4.11 范围

本切片关闭 W1.4.6 留下的 `retry_policy.model_retries` 缺口：Pydantic AI SDK `AgentRetries` 不能表达 model retry，因此由 PydanticAIAdapter 在 session 边界执行 adapter-owned retry。

- `_sdk_agent_retries()` 继续只向 SDK 投影 `tools/output`，不把 `model_retries` 传给 Pydantic AI SDK。
- `run()` 对 retryable model/session exceptions 执行 `retry_policy.model_retries` 次重试；retry 前关闭并丢弃失败 session，重新创建 session，避免复用不确定状态。
- retryable model/session exceptions 包括 Python `TimeoutError` / `ConnectionError`、常见 HTTP/API transport exception class name，以及 HTTP status `408` / `429` / `5xx`；其它 `4xx` 模型 HTTP 错误按 `AA_RUN_STREAM_INTERRUPTED` 失败但不重试。
- Agent / toolset 构造与 registry factory 失败不进入 model retry，避免把工具面配置或 MCP/Skill registry 问题误判为模型请求瞬态失败。
- 已经发出任何 raw stream event 后发生的异常不重试，避免前端收到 partial delta 后 adapter 重放整次请求导致重复输出。
- 重试耗尽后以 `AA_RUN_STREAM_INTERRUPTED` 失败，payload 记录 `model_retries`、`model_retry_attempts` 与 `model_retries_exhausted=true`。

边界说明：W1.4.11 不实现 `retry_policy.backoff` 的真实等待策略，不处理 `escalation_after` / ModelRouter 升级，不实现 per-tool retry map、成本预算/扣减、真实 SDK 上游 `cancel()` <=5s 实测、真实内置工具 / SkillRegistry / MCP lifecycle，也不推进 ClaudeCodeAdapter 真实 SDK/session。
