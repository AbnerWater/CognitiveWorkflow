## W1.3.1 范围

本切片只建立 Runtime sidecar 的 API 基础面，不实现 Engine、持久化、SSE replay 或 Adapter：

- `RuntimeSettings` 固定 `127.0.0.1` 绑定，要求 `CW_RUNTIME_AUTH_TOKEN` 注入，符合 `http_sse.md` D-API-1 / D-API-2。
- `ErrorEnvelope` 对齐 `http_sse.md §1.5` 字段：`schema_version / error_code / message / details / cw_failure_type / retry_after_ms / trace_id`。
- `validate_bearer_authorization()` 对 `Authorization: Bearer <token>` 做常量时间比较；缺失或错误 token 返回 `AUTH_FORBIDDEN`。
- `create_app()` 通过 runtime extra 动态加载 FastAPI，避免默认 `cw_runtime` import 依赖 runtime extras。
- 已落地 system 端点基础响应：
  - `GET /cw/v1/system/info`
  - `GET /cw/v1/system/health`
  - `GET /cw/v1/system/capabilities`
  - `POST /cw/v1/system/shutdown`
- CLI 启动时绑定 `127.0.0.1:<port>`，成功监听后输出 `READY <port>`；缺 token 或缺 runtime extra 时以错误退出，不输出假 READY。

## W1.3.2 范围

本切片建立项目级 Runtime Harness 初始化能力，并接入项目创建/读取 API：

- `cw_runtime.harness` 新增纯 Python 项目初始化逻辑：
  - 创建 `.agent-workflow/` 骨架和用户可见的 `references/`、`workflow/`、`outputs/`。
  - 写入 `project.json`、`settings.json`、`workflow.flow.json`、`workflow_history.json`、`memory.json`、`references.manifest.json`、`skills.config.json`、`mcp.config.json`、`adapters.config.json`、`manifest_revision.json`。
  - 追加 `.gitignore` 与 `.gitattributes` 中 runtime harness 必需规则。
  - 初始化 Git 仓库，安装最小 pre-commit hook，并提交 `chore(cw): initialize CognitiveWorkflow project <project_id>`。
- `manifest_revision.json` 初始记录 8 个 manifest；`update_manifest_json()` 在 `runtime.lock` 下原子写入并递增 revision。
- `memory.json` 默认禁止通过普通 manifest 写入，保留 D-RH-2 的 `memory_task` / 显式 UI 边界。
- `POST /cw/v1/projects` 接入项目创建；`GET /cw/v1/projects/{project_id}` 返回当前 runtime 进程注册的项目资源。
- `POST /cw/v1/projects` 支持 `Idempotency-Key`：同 key + 同 body 重放结果，不同 body 返回 `IDEMPOTENCY_KEY_BODY_MISMATCH`，TTL 外重用返回 `IDEMPOTENCY_KEY_REUSE_OUTSIDE_WINDOW`。

## W1.3.3 范围

本切片建立 Runtime 的 WorkflowGraph validation / compile boundary，不实现 LangGraph `StateGraph`、节点 Runner 或 Adapter 调用：

- 新增 `cw_runtime.engine`，提供 `load_workflow_graph()`、`validate_workflow_graph_payload()`、`compile_workflow_graph()`、`load_and_compile_workflow()`。
- L1 文件/JSON 入口映射到 `workflow_graph.md §8` 已锁错误码：`WG_L1_INVALID_JSON`、`WG_L1_NODES_NOT_ARRAY`、`WG_L1_EDGES_NOT_ARRAY`。
- L2 schema 与节点/边声明一致性继续复用 `cw_schemas.WorkflowGraph`，runtime 保留 `WG_*` / `NC_*` custom error code，并把常见 Pydantic discriminator / enum 错误映射回 spec 错误码。
- `compile_workflow_graph()` 输出内部 `EngineWorkflowIR`，并把 `evaluation_task` 的 pass/fail、`repair_task` 的 repair/retry、`human_checkpoint` 的 decision routes 合并为确定性 Engine 边。
- L3 编译校验覆盖 orphan、unreachable、fail dead-end、uncontrolled loop；合法 retry loop 需要 `retry/loop` 边以及 evaluation `max_retry` guard。
- L4 可行性校验覆盖 enabled node types、global context refs、ModelProfile、Skill、MCP server 外部注册表，不在 schema 层强行实现运行时状态。
- 默认启用节点类型为 Phase 1 runtime 范围：`start / end / execution_task / evaluation_task / repair_task / human_checkpoint`；`tool_task / memory_task / subflow` 仍由 L4 阻止。

边界说明：W1.3.3 只完成 Graph → Engine IR / LangGraph adapter seam；`workflow_graph.md §7` 与 ADR-0004 要求的 WorkflowGraph → LangGraph `StateGraph` 一对一编译仍属于后续 M1.3 Engine/Runner 完成条件，不能用本切片替代最终 StateGraph compiler。

## W1.3.3 审查结果

- 独立 subagent 初审结论：`FAIL`，指出 5 项问题：W1.3.3 scope 与最终 LangGraph StateGraph compiler 目标容易混淆；runtime 错误码归一化丢失 `NC_*` / `WG_L2_METADATA_NOT_NAMESPACED`；human route 合并会丢 decision key；非 LLM 节点的 contract model policy 被 L4 model registry 错误检查；repair-task 回流把 `repair` 边当作 `retry` 边兼容。
- 已修复：文档显式声明本切片不是最终 LangGraph StateGraph compiler；`WorkflowValidationError` 保留 `WG_*` / `NC_*` L1-L4 custom code；human route 按 `route_key` 合并；节点级 model registry 仅检查 execution/evaluation/repair；repair-task 回流只接受/生成 `retry` 语义。
- 回归测试新增覆盖：schema custom error code passthrough、human decision route key、repair vs retry 语义、non-LLM contract model policy。
- 独立 subagent 复核结论：`CONDITIONAL_PASS`；代码级阻塞已关闭，条件是完整 WorkflowGraph → LangGraph `StateGraph` compiler 仍为后续 M1.3 完成项。
