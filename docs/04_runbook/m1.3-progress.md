## W1.3.1 范围

本切片只建立 Runtime sidecar 的 API 基础面，不实现 Engine、持久化、SSE replay 或 Adapter：

- `RuntimeSettings` 固定 `127.0.0.1` 绑定，要求 `CW_RUNTIME_AUTH_TOKEN` 注入，符合 `http_sse.md` D-API-1 / D-API-2。
- `ErrorEnvelope` 对齐 `http_sse.md §1.5` 字段：`schema_version / error_code / message / details / cw_failure_type / retry_after_ms / trace_id`。
- `validate_bearer_authorization()` 对 `Authorization: Bearer <token>` 做常量时间比较；缺失或错误 token 返回 `AUTH_FORBIDDEN`。
- `create_app()` 通过 runtime extra 动态加载 FastAPI，避免默认 `cw_runtime` import 依赖 runtime extras。
- 已落地 system 端点基础响应：
  - `GET /cw/v1/system/info`
  - `GET /cw/v1/system/health`
  - `GET /cw/v1/system/capabilities`
  - `POST /cw/v1/system/shutdown`
- CLI 启动时绑定 `127.0.0.1:<port>`，成功监听后输出 `READY <port>`；缺 token 或缺 runtime extra 时以错误退出，不输出假 READY。

## W1.3.3 范围

本切片建立 Runtime 的 WorkflowGraph validation / compile boundary，不实现 LangGraph `StateGraph`、节点 Runner 或 Adapter 调用：

- 新增 `cw_runtime.engine`，提供 `load_workflow_graph()`、`validate_workflow_graph_payload()`、`compile_workflow_graph()`、`load_and_compile_workflow()`。
- L1 文件/JSON 入口映射到 `workflow_graph.md §8` 已锁错误码：`WG_L1_INVALID_JSON`、`WG_L1_NODES_NOT_ARRAY`、`WG_L1_EDGES_NOT_ARRAY`。
- L2 schema 与节点/边声明一致性继续复用 `cw_schemas.WorkflowGraph`，runtime 保留 `WG_*` / `NC_*` custom error code，并把常见 Pydantic discriminator / enum 错误映射回 spec 错误码。
- `compile_workflow_graph()` 输出内部 `EngineWorkflowIR`，并把 `evaluation_task` 的 pass/fail、`repair_task` 的 repair/retry、`human_checkpoint` 的 decision routes 合并为确定性 Engine 边。
- L3 编译校验覆盖 orphan、unreachable、fail dead-end、uncontrolled loop；合法 retry loop 需要 `retry/loop` 边以及 evaluation `max_retry` guard。
- L4 可行性校验覆盖 enabled node types、global context refs、ModelProfile、Skill、MCP server 外部注册表，不在 schema 层强行实现运行时状态。
- 默认启用节点类型为 Phase 1 runtime 范围：`start / end / execution_task / evaluation_task / repair_task / human_checkpoint`；`tool_task / memory_task / subflow` 仍由 L4 阻止。

边界说明：W1.3.3 只完成 Graph → Engine IR / LangGraph adapter seam；`workflow_graph.md §7` 与 ADR-0004 要求的 WorkflowGraph → LangGraph `StateGraph` 一对一编译仍属于后续 M1.3 Engine/Runner 完成条件，不能用本切片替代最终 StateGraph compiler。

## W1.3.4 范围

本切片建立 WorkflowRun lifecycle 与 StreamEvent bus 的服务端/落盘边界，不执行节点、不启动 Adapter、不推进 LangGraph：

- 新增 `cw_runtime.runs`，提供 `create_workflow_run()`、`read_workflow_run()`、`pause_workflow_run()`、`resume_workflow_run()`、`cancel_workflow_run()`、`list_stream_events()`、`stream_sse_events()`、`format_sse_event()`。
- `POST /cw/v1/workflows/{workflow_id}/run` 创建 `runs/<run_id>/`，写入 `run.json`、空的 attempts/evaluations/repairs/decisions/usage/metrics jsonl、`skill_lock.json`、`mcp_lock.json`，并发出 `run.started`。
- `GET /cw/v1/runs/{run_id}` 返回 `run.json` 投影；`POST /cw/v1/workflows/{workflow_id}/pause|resume|cancel` 驱动当前 active Run 状态迁移；`POST /cw/v1/runs/{run_id}/cancel` 保留为 run-level 取消入口。
- `GET /cw/v1/runs/{run_id}/stream` 与 `/observability/runs/{run_id}/stream` 使用 live `StreamingResponse`，按 `stream_event.md` D-SE-2 输出 SSE 帧，支持 `Last-Event-ID` replay、`since_seq / until_seq`、`category / level` 过滤与 `system.heartbeat`。
- 创建 Run 时执行 W1.3.3 的 WorkflowGraph L1-L4 validation / Engine IR compile boundary，并阻止同一 Workflow 的并发非终态 Run（`WR_CONCURRENT_RUN_FORBIDDEN`）。
- Run 创建、状态迁移与 StreamEvent append 在 `runtime.lock` 内串行化；generic resume 只允许 `paused -> running`，`waiting_user / repairing` 仍由后续 `decision.resolved / patch.applied` 路径触发。
- Project 初始化追加 `.agent-workflow/runs/*/stream-events/` 与 planning stream-events 的 git ignore/pre-commit 阻断，符合 D-RH-10。
- 副作用 POST 复用 API idempotency cache：同 path + 同 `Idempotency-Key` + 同 body 在 24h 内重放响应。

边界说明：W1.3.4 是 Run 壳层和事件总线，不生成 NodeAttempt 内容，不写 EvaluationResult/RepairPatch/HumanDecision 业务记录，也不创建 git run snapshot/tag；这些由 W1.3.5+ Runner、Repair/HITL 与 Persistence slice 补齐。

## W1.3.5 范围

本切片建立确定性 Node runner foundation，让 W1.3.4 的 Run lifecycle / StreamEvent bus 能推进最小节点闭环；不调用真实模型、不引入 LangGraph 运行依赖、不实现 ContextBuilder/EvidenceBuilder 或 Human Decision resolve：

- 新增 `cw_runtime.runner.advance_workflow_run()`，从 `run.json.current_node_ids` 取单个当前节点，在 `runtime.lock` 内推进一步。
- 覆盖 Phase 1 节点类型：`start / end / execution_task / evaluation_task / repair_task / human_checkpoint`；`tool_task / memory_task / subflow` 仍不在本切片执行范围。
- `execution_task` 使用显式 deterministic `ExecutionAdvanceInput` 生成 `NodeAttempt`，写入 `attempts.jsonl`，并写入最小有效 `context_packs/<id>.json` 与 `execution_packs/<id>.json` foundation stub，避免 attempt 引用悬空。
- `evaluation_task` 使用 deterministic `EvaluationAdvanceInput` 生成完整 `EvaluationResult`，写入 `evaluations.jsonl`，按 `pass_to_next / repair_with_patch / human_checkpoint / retry_same` 的可判定路由推进。
- `repair_task` 使用 deterministic `RepairAdvanceInput` 生成 `RepairPatch`，写入 `repairs.jsonl`，登记目标节点 pending prompt overlay；目标 execution retry attempt 创建时写入 `overlays/<target_attempt_id>.json`，并挂到该目标 attempt 的 `ExecutionPack.effective_prompt_overlay` 与 `NodeAttempt.effective_prompt_overlay_ref`。
- `human_checkpoint` 进入 `waiting_user`，发出 `human.gate_required` + `run.paused`，不实现 `decision.resolved`；`waiting_user -> running` 仍留给后续 HITL resolve 切片。
- 节点运行态与 attempt 计数放在 `run.json.metadata.cw.node_states / attempt_counts / last_attempt_ids / last_evaluation_by_target / last_patch_ids`，不扩展 `run.json` 顶层字段。
- StreamEvent 严格使用已锁事件类型：`node.state_changed`、`attempt.started/completed/failed`、`evaluation.started/completed`、`repair.started/patch_proposed/patch_applied`、`human.gate_required`、`run.paused/completed/failed`。

边界说明：W1.3.5 是确定性 runner foundation，不等同于最终 WorkflowGraph → LangGraph `StateGraph` 执行器；真实 Adapter 调用、Context/Evidence pack 构建、Human Decision resolve、SQLite/OTel/git snapshot 都仍属于 W1.3.6+ / W1.3.7。

## W1.3.6 范围

本切片把 W1.3.5 的 foundation stub pack 替换为 deterministic static ContextBuilder / EvidenceBuilder Phase 1，不调用真实模型、不读取 ReferenceLibrary 索引、不调用工具、不写缓存数据库：

- 新增 `cw_runtime.builders`，提供 `build_static_attempt_packs()`，一次性构建 `EvidencePack | None`、`ContextPack` 与 `ExecutionPack`。
- Static ContextBuilder 覆盖 `node_goal`、`context_requirements.kind=static_text`、`context_requirements.kind=user_input`，从 `run.metadata.cw.initial_input` 读取确定性输入，并填充 `template_inputs.deps`。
- Static EvidenceBuilder 仅在 `NodeContract.evidence_requirements` 非空时运行；Phase 1 只接受 `initial_input.user_assertions` / `initial_input.evidence` 的显式用户断言作为 `user_input` evidence。
- 必填 evidence requirement 无可用用户断言时以 `EP_BUILD_REQUIREMENT_UNRESOLVED` 构建期失败；不生成空 EvidencePack，不伪造证据。
- EvidencePack 构建成功后，ContextBuilder 将每条 Evidence 投影为 `kind=evidence` 的 `ContextFragment`，并在 `deps.evidence` 中提供引用列表。
- Runner 在 attempt 启动前写入 `evidence_packs/<id>.json`、`context_packs/<id>.json`、`execution_packs/<id>.json`，并发出 `context.build_started`、`evidence.build_completed`、`context.build_completed`。
- `NodeAttempt.evidence_pack_id` 与 `EvaluationResult.provenance.evidence_pack_id` 从本次 bundle 透传，不再硬编码为空。

边界说明：W1.3.6 不是最终 MCCL 检索实现。`reference`、`project_memory`、`upstream_artifact`、ReflectionMemory 注入、压缩摘要、cache SQLite、敏感 secure 旁路、真实 EvidenceBuilder retrieve/rerank/consolidate 与 CitationChecker 仍属于后续 M1.3/W1.4 切片。
